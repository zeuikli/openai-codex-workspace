## ADOPTION-GUIDE 適當性評估（本人 Before P2 #7 提案之落地）

**整體判斷：方向正確、落地工程紮實，但仍非 Body 替代物。**

**適當之處**：
- §0 30 秒誠實聲明把 v4 forward canonical / `[E]` ≠ enforce / 20/22 point estimate / ChatGPT uncalibrated 四項前置，採用者無法跳過。
- §1 決策樹四列（只要認知協議 / Claude 類宿主 / ChatGPT / 自建 API agent）涵蓋外部採用主要分岔，每列給「讀什麼 / 貼什麼 / 完成上限」三欄，是落地路徑首次機械化。
- §2 四欄高風險子集對照表（CORE 條文 → fixture → Body 狀態 → backlog）正是本人 Before 提案的 P2 #7 (a)。
- §3 EXPORT 失真點 ↔ CORE §7 雙向導讀是 Before 提案的 P2 #7 (b)。
- §4.1 / §4.2 人讀 + 機讀 JSON skeleton（含 `schema_version` / `provenance` / `recalibration_trigger`）填補 Before L2 缺範本缺口。
- §5 ban list 與 §6 week-1 驗收清單（選 3 fixtures、fresh context、外部 verifier、記錄 n、標 uncalibrated、不 pool runtime）是落地防誤用最具體的一段。

**待改善（落地後仍須補）**：
1. **§2 表只列高風險子集，但未明示「完整矩陣見 SPEC §2.2 / EVAL-BASELINE §1」的跳轉是替代而非補充**——採用者可能誤以為這十行就是全部 Body 對照。建議在表頭加一句「本表是入門子集；落地任何 `[E]` 條文前須先讀 SPEC §2.2 完整矩陣」。
2. **§4.2 機讀 JSON skeleton 的 `global_thresholds` 與 PROFILES-v4.md §2 的 n_min_per_cell / loop_max_iter / no_progress_threshold / budget_cap 對齊未明示**——採用者填 placeholder 時須自行對照 PROFILES §2 數字來源。建議在 skeleton 註解加「數字來源見 PROFILES-v4.md §2；本 skeleton 不含預設值」。
3. **§6 week-1 驗收清單未明確連結 #2c held-out rerun 協議**——清單要求記錄 n、標 uncalibrated，但採用者跑自家 fixture 時，與本 workspace #2c 協議的關係（是否可回饋數字、是否可外推）未說明。建議加一句「採用者自家 baseline 與本 workspace #2c 不互相背書；數字分開記帳」。

**結論**：ADOPTION-GUIDE 是落地可行性的實質推進，把 Before P2 #7 提案具體化；但它本身是 adopt-time guide，不改變 Body 狀態——採用者仍須自建 hook / runner / verifier 才能把 `[E]` 從 advisory 升格。
